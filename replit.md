# CareMetric AI Intelligence

## Overview
CareMetric AI Intelligence is a React/Vite frontend application designed for healthcare provider intelligence, CRM, and analytics. It serves as a comprehensive data platform enabling provider management, lead generation, email campaigns, CMS analytics, and AI-driven insights to enhance healthcare provider strategies and outreach. The project's vision is to streamline healthcare data analysis and provider engagement, offering a robust toolset for market advantage and operational efficiency.

## System Architecture
The application is built with a React 18 + Vite 6 frontend, styled using TailwindCSS and Radix UI components, specifically leveraging `shadcn/ui`. State management is handled by TanStack React Query, and routing uses React Router v7. The backend is an Express.js API (port 3001) providing generic entity CRUD operations. Data persistence is managed by PostgreSQL with Drizzle ORM, encompassing 52 distinct tables. Authentication is JWT-based, utilizing bcrypt for password hashing. The system integrates AI capabilities via Anthropic Claude, email services through SendGrid, and file uploads via Multer. Frontend API client (`src/api/client.js`, re-exported as `base44` from `src/api/base44Client.js`) mirrors the Base44 SDK interface for consistent interaction with backend entities, integrations, and functions. In development, Vite (port 5000) proxies `/api` to the Express server; the frontend always talks to the same-origin `/api`.

The system spans three layers: (1) the **React SPA** (`src/`), with pages auto-registered in `src/pages.config.js`; (2) the **Express API** (`server/`); and (3) a layer of **Base44 Deno edge functions** (`base44/functions/`) that handle scheduled imports, maintenance fan-out, and import lifecycle hooks on the Base44 platform.

**Key UI/UX decisions include:**
- **Dark Theme:** Strict adherence to a dark theme palette: `text-slate-300/400`, `bg-slate-800/900`. Accent colors use `bg-*-900/30 text-*-400 border-*-500/30`.
- **Responsive Design:** Page wrappers utilize `p-4 sm:p-6 lg:p-8 max-w-[1400px] mx-auto`, with grids collapsing on smaller screens and tables supporting horizontal overflow.

**Core Technical Implementations and Features:**
- **Provider Management:** Comprehensive tools for managing providers, locations, and territory mapping.
- **Facility Pages:** Unified detail pages for 8 facility groups: Hospitals, Home Health, Hospice, SNF, IRF, LTCH, DME, Community Health (combined FQHC + RHC). Each group has a searchable list page (with state filter) and a detail page consolidating all imported CMS data per facility. Backend: `server/functions/facilityDetail.ts` (getFacilityDetail, listFacilities, handleGetProviderCMSData). Pages: `Hospitals.jsx`, `HomeHealthAgencies.jsx`, `Hospices.jsx`, `NursingHomes.jsx`, `InpatientRehab.jsx`, `LongTermCare.jsx`, `DMESuppliers.jsx`, `CommunityHealthCenters.jsx`, `FacilityDetail.jsx`. **FacilityDetail** shows all raw_data columns (searchable, expandable table), 6 KPI summary cards, and a rich Provider Intelligence card (email, phone, lead score, taxonomies, status, "Full Provider Profile" CTA) when the facility provider_id matches a known NPI. **ProviderDetail** uses `base44.functions.invoke('getProviderCMSData')` for CMS data (not raw fetch), and shows LinkedFacilitiesCard on the Overview tab for quick facility cross-reference. LinkedFacilitiesCard handles all 8 facility groups for navigation. DB indexes: `idx_medfac_type_name`, `idx_medfac_provider_id`, `idx_medfac_type_state`. Listing queries use a single "primary type" per group for performance; `community_health` queries both `fqhc_enrollments` and `rural_health_clinic_enrollments`.
- **Provider CMS Data:** `handleGetProviderCMSData` returns MIPS, clinician profile, group practice, physician utilization, Part D prescribing, DME referrals, drug spending, facility affiliations, telehealth trends, and ACO network data. Data is grouped into `clinician`, `utilization_cms`, and `network` (ACO) categories. Displayed via `MIPSPerformanceCard` and `ProviderCMSDataCard` on ProviderDetail's Clinical and Network tabs. ACO data (participants, organizations, financial results) appears in a dedicated network section. `cmsImportTypes.js` contains labels for all 90+ configured dataset types. FACILITY_TYPE_GROUPS maps ~65 dataset types to 8 facility groups, ensuring all quality, payment, staffing, ownership, and VBP data flows into the correct facility detail page.
- **Sales & Outreach:** Functionality for lead lists, provider intelligence, and multi-channel campaigns.
- **Analytics:** CMS data analysis, network insights, and custom reporting capabilities, including materialized views for performance.
- **AI Assistant & Enrichment:** AI-powered tools for data enrichment and insights.
- **Data Import System:** Robust import mechanisms for Medicare/CMS datasets, NPPES flat files, and ZIP imports with pause/resume, error handling, and stall detection. **DataCenter** (`DataCenter.jsx`) is the import entry point with 9 category cards (Provider Data, Claims & Utilization, Hospitals & Quality, Home Health, Hospice, SNF & Nursing Homes, IRF/LTCH/DME, Enrollment & Market, Drug Spending) covering 90+ dataset types. `DataDestinationMap.jsx` maps every import type to its "View Data" destination page (facility group pages, Utilization, Providers, Organizations, CMSAnalytics, etc.) for post-import navigation. `ImportCategoryCards.jsx` defines the importable categories. `ImportMonitoring.jsx` provides real-time monitoring with 7 tabs (Monitor, History, Rules, Alerts, AI, AI Manager, Performance).
- **Security:** JWT-based authentication with role-based access control (adminOnly middleware), sensitive field stripping, and file upload size limits. User entity is explicitly excluded from generic CRUD for security. **Production secrets required:** `JWT_SECRET` (server hard-fails on startup if missing in production); `ADMIN_EMAIL` + `ADMIN_PASSWORD` (used to seed the first admin — production skips seeding with a warning if `ADMIN_PASSWORD` is unset; dev falls back to built-in defaults). **Entity router ACL:** sensitive entities (`ReconciliationSettings`, `ApiInteractionLog`, `NPPESCrawlerConfig`, `CMSApiConnector`, `ImportScheduleConfig`, `AuditEvent`, `ErrorReport`, `BackgroundTask`, `ImportBatch`, `ImportValidationRule`, `DataCleaningRule`, `ColumnMappingRule`, `ReconciliationJob`, `ProviderReconciliation`, `NPPESQueueItem`) are admin-only for both read and write. A whitelist of collaborative entities (lead lists, campaigns, scoring rules, custom reports, saved filters, etc.) allows non-admin writes; everything else requires admin. Internal DB error messages are no longer leaked to clients in production. ID parameters use strict integer validation (`/^-?\d+$/` + `Number.isSafeInteger`).
- **NPPES Crawler:** A dedicated crawler for the NPPES registry with actions for status, batch control, and queue processing.
- **Email Search Bot:** An AI-powered bot (Claude-based) for email discovery, validation, and automated saving to provider records. **Optimized for throughput:** single combined Claude API call (search + validate in one prompt, was 2 calls), 5 concurrent worker threads per batch, 300ms inter-provider delay (was 1500ms), default batch size 10. `providerIntelligenceBot.ts` also uses 5 concurrent workers and 300ms delay. Auto-resumes on server restart: if a task was in "processing" status, it resumes it; if the last task was "failed" (e.g. from a crash), it auto-starts a new task as long as unsearched providers remain and the last task wasn't explicitly cancelled. Batch size is always clamped to [1, 50]. Multiple orphaned "processing" tasks are reconciled (one resumed, extras marked failed). Workers check for cancellation every 10 providers for responsive stop behavior.
- **Data Quality Scan:** Utilizes sampling for performance, runs 13 quality rules, and generates alerts with Claude AI integration for analysis and fixes.
- **Backend Functions:** The Express function dispatcher (`server/routes/functions.ts`) exposes 53 functions, including optimized dashboard statistics, data health metrics, CMS dataset catalog, CMS analytics, referral network data, territory data, facility/provider detail, county intelligence, comprehensive reports, import triggers, NPPES crawler control, email/intel/enrichment bots, and campaign tooling. **Authorization is default-deny:** a `PUBLIC_FUNCTIONS` allowlist (read-only/analytics/status functions) is callable by any authenticated user; every other function (imports, crawler control, token-spending AI enrichment, email sending, destructive maintenance) is admin-only. Adding a new function is therefore an explicit security decision. **Dashboard stats** (`getDashboardStats`) uses raw pool connection with `SET statement_timeout = '5s'`, `pg_class` estimates for total counts, and `LIMIT`-based sampling (2000 providers, 10000 locations) for email/phone stats — extrapolated to full scale. Results cached 5 minutes. Responds in ~400ms cold, ~60ms cached.
- **Base44 Edge Functions & Scheduled Maintenance:** `base44/functions/` holds 26 Base44 **Deno** edge functions (importers, hooks, workers, AI helpers) that run on the Base44 platform — distinct from the Express backend. `runScheduledImports` is the cron driver: each tick it claims due `ImportScheduleConfig` rows (lease-based, `CLAIM_LEASE_MS`), runs them, computes the next run, and then **fans out the maintenance workers** (`autoResumePausedImports`, `autoRetryFailedImports`, `manageCrawlerRetries`, `cancelStalledImports`). Driving maintenance from this reliably-scheduled function (rather than separate, easy-to-misconfigure platform crons) means a single cron outage is the only failure mode. Each worker is idempotent and self-throttling. Import lifecycle hooks (`onImportBatchCompleted`, `onImportBatchFailed`, `onRuleCreated`) and per-state NPPES supervision (`superviseNPPESCrawler`) also live here. Pure scheduling/retry logic is factored into `helpers.ts` per function so it can be unit-tested from Node without booting Deno.
- **Auto-Retry & Import Health:** Failed import batches whose error is transient are retried automatically. `autoRetryFailedImports/helpers.ts` classifies retryable errors (network/HTTP 5xx/429/rate-limit/connection keywords), tracks attempt counts (capped at `MAX_AUTO_RETRY_ATTEMPTS`), and schedules the next attempt with exponential backoff (capped at `RETRY_BACKOFF_CAP_HOURS`). The frontend mirrors this logic in `src/components/imports/retryStatus.js` and computes health rollups in `healthMetrics.js` (success-rate-by-day, top error categories, retry-pipeline buckets, unhealthy schedules). `MaintenanceHealthPanel.jsx` (embedded in `ImportMonitoring.jsx`) polls the latest `maintenance_fanout` `AuditEvent` heartbeat and warns when maintenance is stale, partial, or the cron is unwired. `ImportHealthDashboard.jsx` renders the 30-day outcome chart, error breakdown, retry pipeline, and schedule-health table.
- **County Intelligence Page:** `CountyIntelligence.jsx` — state selector + city text filter, shows provider counts, specialty breakdown bars, referral flows, provider/facility lists. Backend: `server/functions/countyIntelligence.ts` uses raw `pool.connect()` queries with `statement_timeout` for performance on 12M+ provider_locations table. Uses LIMIT 200 NPI sampling + parallel IN-clause queries. States list is hardcoded (DISTINCT on 12M rows is too slow). Nav entry: "County Intel" in Analytics section.
- **Comprehensive Report:** `ComprehensiveReport.jsx` — lazy-loaded, printable, collapsible sections showing all provider data (locations, taxonomies, utilization, referrals, affiliations, lead scores, enrichments). Backend: `server/functions/comprehensiveReport.ts` uses single pooled connection with `statement_timeout`. Integrated as "Full Report" tab (#7) in ProviderDetail and inline button in FacilityDetail.
- **Relationship Cards:** `ReferralPartnersCard.jsx` (sent/received tabs with bar charts) in ProviderDetail Network tab. `AffiliatedProvidersCard.jsx` (uses ProviderAffiliation entity) in FacilityDetail.
- **DB Performance Notes:** `provider_locations` (12M rows), `providers` (6M rows), `medicare_facilities` (46M rows). Avoid DISTINCT/GROUP BY on provider_locations or full-table scans on medicare_facilities without proper indexes. The `medicare_facilities` table lacks a standalone `state` index (only has `(facility_type, state)` composite). Referrals table (`cms_referrals`) only has NPI index, not `referred_to_npi` index. Use IN-clause with pre-fetched NPI lists instead of JOINs/subqueries.

## External Dependencies

- **AI:** Anthropic Claude (via `/api/integrations/ai/invoke`). Models centralized in `server/lib/aiModels.ts`: **Sonnet 4.6** (`claude-sonnet-4-6`) for conversational assistants, multi-step reasoning, and analytical/structured output (Data Quality Assistant, Pattern Analysis, AI Import Manager chat, default `/ai/invoke`); **Haiku 4.5** (`claude-haiku-4-5-20251001`) for high-volume bulk worker loops (Email Search Bot, Provider Intelligence Bot, third-party enrichment, custom data-point lookup); **Opus 4.7** (`claude-opus-4-7`) reserved for the heaviest reasoning tasks. All models can be overridden via `CLAUDE_MODEL_SONNET` / `CLAUDE_MODEL_HAIKU` / `CLAUDE_MODEL_OPUS` env vars without code changes.
- **Email:** SendGrid (via `/api/integrations/email/send`)
- **Database:** PostgreSQL (with Drizzle ORM). Migrations are in `./drizzle/` (generated via `npm run db:generate`). The initial migration uses `IF NOT EXISTS` to be safe against existing production tables. For development, `npm run db:push` syncs schema directly. For production deployments, use `npm run db:migrate`.
- **File Upload:** Multer (for `/api/integrations/file/upload`)
- **Edge Runtime:** Base44 platform (Deno) for `base44/functions/` — uses `npm:@base44/sdk` and `npm:`-prefixed imports; not part of the Node/Express runtime.
- **Testing:** Vitest + React Testing Library + jsdom (dev dependencies); CI via GitHub Actions.
